汉堡包、爬山车、截止日期

22/9/11 17：44斜阳，阳台窗边

    从阳台边醒来，望见远方如大雾一般若隐若现的淡蓝色海岸线。脚下车水马龙，好不热闹。仔细一看，这并非是什么阳台，而是活脱脱的一座摩天轮。摩天轮上只有一个船舱，但我无论如何都想不起自己是怎样上来的。
    太阳被视野右侧高高瘦瘦的西式居民房所遮挡，卡其色的墙面与深蓝色的穹顶边缘泛着柔和的光线，散射在天空中，就像蛋黄奶油的馅料一样。
    摩天轮还在缓慢爬升，大约在时钟上十刻的地方。虽然视野正在逐渐变得敞亮，却依然没有能够看清太阳的迹象。又或者那只是我此刻心理的假象，但在那之前，寒冷和饥饿可能会让我看不到下一轮太阳。我醒悟过来，慌忙地从随身斜跨的蓝色运动包中寻找食品。由于平日里的坏习惯，里面到处充满了开过的塑料袋。只剩下一个用纸包着的汉堡是完好无损的。里头是两片干瘪的面包，夹着一块干柴的混合肉片，缝隙间隐约可见芝士和酸黄瓜，拿在手上也没有丝毫温度，如果现在是下午的话，至少是清晨买来的吧。
    没有任何犹豫的余地，我胡乱地将其往嘴里乱塞一通，芝士中透露着酸涩感和一丝番茄酱的甜味。没有饮料的话，实在有些令人生腻，是标准的工业流水线口味，打上不上不下的品牌的价格，便可用来度过枯燥无味的一天。
    既然我是八九个小时前买来的这汉堡，那中间做了些什么，又是如何到这摩天轮顶上，我还是难以找到蛛丝马迹。虽然我一向认为自己是一个理性的人，至少足以完成理工科学士。但照着我的人生经历来说，可能是为了逃避某些事情，又或是在紧急情况下被人坑蒙到这里来也未尝得知。
    把时间推到更久以前，依稀能够记得昨天晚上，在木栈桥道的灯光下。与一位友人相谈甚欢。从近况，到文学、爱好、计算性能等等，几乎无所不谈。友人穿着不对称的红白大衣，左手是束口的红色布料，从胸口到右手则是米黄色宽松的长袖，大衣之下是一条宽大的灯笼裤。本来在都市中应该也是引人注目的奇异造型，但是这一身浅色色调在黑夜中反而十分的低调。
    我们聊到了他未发的新诗，新诗自然提到截稿日，而他也一如既往一副毫不关心的样子，好像世界的截止日期都已将至，一篇小诗的事情就像一个玩具恐龙在三百米高的克苏鲁水怪面前一般平静而渺小。现在，这个世界到处都在变得模糊，就像漫起大雾，但实际上却是天气晴朗，秋高气爽。楼下的车流来来往往，车笛的啼叫，轮胎摩擦地板的声音，引擎的轰鸣，却越发让我感到平静。就像四周的居住区，别墅区，大约已经住满了人，却没有一户人家开灯。不往公路上看的话，仿佛世界已经停止了运转。在模糊中，好像忘记掉未来的截止日期，忘记掉自己做过什么事情，反而会比较幸福。在白雾中，我不再思考自己为何来到这里，而是琢磨起昨晚灯火阑珊处的诗篇。

git reset --hard  : 1.移动本地版本库的指针；2.重置暂存区；3.重置工作区。

配置文件

.gitignore 用于配置不需要git控制的文件 支持正则表达式

文本规范

.gitignore 的格式规范如下：
• 所有空行或者以 # 开头的行都会被 Git 忽略。
• 可以使用标准的 glob 模式匹配，它会递归地应用在整个工作区中。
• 匹配模式可以以（/）开头防止递归。
• 匹配模式可以以（/）结尾指定目录。
• 要忽略指定模式以外的文件或目录，可以在模式前加上叹号（!）取反。
# 忽略所有的 .a 文件

示例

*.a
# 但跟踪所有的 lib.a，即便你在前面忽略了 .a 文件
!lib.a
# 只忽略当前目录下的 TODO 文件，而不忽略 subdir/TODO
/TODO
# 忽略任何目录下名为 build 的文件夹
build/
# 忽略 doc/notes.txt，但不忽略 doc/server/arch.txt
doc/*.txt
# 忽略 doc/ 目录及其所有子目录下的 .pdf 文件
doc/**/*.pdf

磁盘阵列（Redundant Arrays of Independent Disks，RAID）

在单机时代，采用单块磁盘进行数据存储和读写的方式，由于寻址和读写的时间消耗，导致I/O性能非常低，且存储容量还会受到限制。另外，单块磁盘极其容易出现物理故障，经常导致数据的丢失。因此大家就在想，有没有一种办法将多块独立的磁盘结合在一起组成一个技术方案，来提高数据的可靠性和I/O性能呢。

概念

RAID即独立磁盘冗余阵列，简称为「磁盘阵列」，用多个独立的磁盘组成在一起形成一个大的磁盘系统，从而实现比单块磁盘更好的存储性能和更高的可靠性。

常见方案分类

RAID0

RAID0 简单的将多块磁盘组合在一起形成一个大容量的存储。写数据的时候，将数据分为N份，以独立的方式实现N块磁盘的读写，那么这N份数据会同时并发的写到磁盘中，因此执行性能非常的高.理论上为N倍。

缺点

它并不提供数据校验或冗余备份，因此一旦某块磁盘损坏了，数据就直接丢失，无法恢复了。因此RAID0就不可能用于高要求的业务中，但可以用在对可靠性要求不高，对读写性能要求高的场景中。

RAID 1

RAID1 是磁盘阵列中单位成本最高的一种方式。
两块磁盘无差别备份，空间只有R0的一半。

RAID 3

在R0的基础上，一个磁盘作奇偶校验。
如果有磁盘坏了，就用其他磁盘补上。
缺点是这一块磁盘频繁访问容易损坏。

RAID 5（最常用）

将 存储性能、数据安全、存储成本 兼顾的一种方案
将奇偶校验信息分布到各个磁盘上
RAID5校验位算法原理：P = D1 xor D2 xor D3 … xor Dn （D1,D2,D3 … Dn为数据块，P为校验，xor为异或运算）

条件：三台以上磁盘，每次只能坏一个硬盘。两个或以上硬盘坏了无法恢复数据。

RAID6

为什么RAID6这么牛呢，因为了双重校验的概念。
RAID6除了每块磁盘上都有同级数据XOR校验区以外，还有针对每个数据块的XOR校验区，这样的话，相当于每个数据块有两个校验保护措施，因此数据的冗余性更高了。

但是RAID6的这种设计也带来了很高的复杂度，虽然数据冗余性好，读取的效率也比较高，但是写数据的性能就很差。因此RAID6在实际环境中应用的比较少。

HDFS

HDFS（Hadoop Distributed File System），是一种分布式文件管理系统

一、性质

优点：

1.高容错，多副本
2.处理数据规模大
3.可以构建在廉价机器上，不怕出问题

HDFS的使用场景：适合一次写入，多次读出的场景。

缺点：

1.不适合低延迟访问
2.不适合存储小文件（占用NN资源，寻址时间超过读取时间）
3.不支持并发写入（一文件一线程
4.不支持随机修改（只能支持追加append

二、组成架构

1.NN

1.管理HDFS命名空间
2.管理副本策略
3.数据块映射
4.处理客户端请求

2.DN

1.接受NN命令，存数据，读写数据

3.Client

1.文件切分，在文件上传HDFS之前完成
2.与NN交互→文件位置
3.与DN交互→存取
4.部分管理功能，如格式化NN
5.部分HDFS增删改查功能

4.2NN

1.定期合并Fsimage和Edits，推送至NN
2.紧急情况下可用于恢复NN（期间的数据可能会损失）
3.考点：不完全是NN的备份
